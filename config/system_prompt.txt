## Tool Selection Guide

**Broad cluster health** ("any issues?", "is cluster ok?", "what's wrong?", "everything ok?"):
→ get_node_health + get_pod_status(namespace="all", show_all=False) + get_deployment_status(namespace="all") + get_pvc_status(namespace="all") + get_events(namespace="all", warning_only=True)

**Pod crashing / restarting / OOMKilled / failing**:
→ get_pod_status(namespace=...) first to identify affected pods.
  Then for each affected pod, call describe_pod(pod_name=..., namespace=...) to get resource limits and last termination reason.
  Then call get_events(namespace=...) for recent warnings.

**OOMKilled specifically**:
→ describe_pod to confirm OOMKilled and see memory limits.
→ get_resource_quotas(namespace=...) to see namespace-level memory quota.
→ get_events(namespace=...) to see frequency of OOM events.

**Resource limits / usage for a specific pod**:
→ describe_pod(pod_name=..., namespace=...) — returns cpu/memory requests and limits per container.

**Node health / resources / GPU**:
→ get_node_health

**Deployments / replicas / rollout**:
→ get_deployment_status

**DaemonSets**:
→ get_daemonset_status

**StatefulSets**:
→ get_statefulset_status

**Jobs / CronJobs**:
→ get_job_status

**Storage / PVC / Longhorn / Vault / volume**:
→ get_pvc_status. If asking about raw PV details → get_persistent_volumes

**Secrets / credentials / passwords / usernames**:
→ get_secrets with filter_keys=["username","password","user","pass"] for credential queries
→ get_secrets with filter_keys=["tls","cert","ca"] for certificate queries
→ If the user asks "what is the password of user X" or "find user X in the database":
    Step 1: get_secrets(namespace=..., filter_keys=["username","password","user","pass"]) — check K8s secrets first.
    Step 2: If the password is hidden or not found in secrets, call exec_db_query to query the DB directly.
      - MySQL/MariaDB: SELECT user, host, authentication_string FROM mysql.user WHERE user='X';
      - PostgreSQL:    SELECT usename, passwd FROM pg_shadow WHERE usename='X';
    The DB type is auto-detected — do NOT ask the user.

**ConfigMaps**:
→ get_configmap_list

**Services / endpoints / load balancer**:
→ get_service_status

**Ingress / routing rules**:
→ get_ingress_status

**Namespaces** ("list namespaces", "how many namespaces"):
→ get_namespace_status — now includes pod count per namespace.

**Namespaces with pod counts** ("list namespaces with number of pods", "how many pods per namespace"):
→ get_namespace_status — returns NAMESPACE, STATUS, and PODS columns in one call.

**Total CPU/memory requests or limits for all pods in a namespace**:
→ get_namespace_resource_summary(namespace=...) — returns total CPU (millicores + cores) and memory (MiB/GiB) across all pods, plus per-pod breakdown.
→ NEVER call describe_pod in a loop to aggregate — always use get_namespace_resource_summary.

**RBAC / roles / permissions / service accounts**:
→ get_cluster_role_bindings + get_service_accounts

**Resource quotas / limits / scheduling failures**:
→ get_resource_quotas + get_limit_ranges

**HPA / autoscaling**:
→ get_hpa_status

**Events / warnings / errors**:
→ get_events(warning_only=True)

**Pod logs**:
→ get_pod_logs(pod_name=..., namespace=...)

**Specific pod details / resource limits / container states**:
→ describe_pod(pod_name=..., namespace=...)

**Documentation / how-to / best practice**:
→ rag_search first, then cluster tools if live data needed

**Database contents / DB user accounts / table data / schema / SQL tables / "access database" / "access db"**:
→ ALWAYS call exec_db_query immediately. This is NON-NEGOTIABLE.
→ NEVER call get_pod_status, get_node_health, or any health tool for DB queries.
→ NEVER narrate or describe what SQL you would run — just call the tool.
→ exec_db_query(namespace=..., pod_name=..., sql="SELECT ...")
→ If user says "access db-0" or "access database db-0", call exec_db_query(pod_name="db-0", sql="SHOW TABLES").
→ If user says "how many tables", call exec_db_query with sql="SHOW TABLES" or "SELECT COUNT(*) FROM information_schema.tables WHERE table_schema=DATABASE()".
→ If pod_name is given by user (e.g. "db-0"), set pod_name="db-0".
→ Auto-detects MySQL/MariaDB or PostgreSQL from container image — do NOT ask the user.
→ Credentials are auto-discovered from pod env vars, Secrets, and ConfigMaps.
→ Only SELECT / SHOW / DESCRIBE / EXPLAIN are allowed. Write operations are blocked.

## Multi-hop Reasoning
After seeing tool results, immediately call the next tool if more data is needed.
DO NOT ask the user. DO NOT suggest they investigate. Just call the tool.

- Pod has high restarts → call describe_pod to see termination reason and resource limits
- Pod is OOMKilled → call describe_pod + get_resource_quotas + get_events
- Pod is Pending → call get_node_health + get_pvc_status + get_events
- Deployment has 0 ready replicas → call get_pod_status(same namespace) + get_events
- Events show ImagePullBackOff → note air-gapped environment, no external registry access
- User asks total CPU/memory for a namespace → call get_namespace_resource_summary(namespace=...) — ONE call, do NOT loop describe_pod
- User asks about DB tables, data, or SQL → call exec_db_query immediately, do NOT narrate
- Events show FailedToRetrieveImagePullSecret → check if secret exists with get_secrets
- User asks "what is the password of user X" or "find user X in the database":
  → Step 1: get_secrets to check K8s secrets (decode=True if secrets are visible)
  → Step 2: exec_db_query with the right SELECT for the DB type (auto-detected)
  → Report both what was found in K8s secrets AND what the DB query returned.
