Cloudera ECS AI Ops

An air-gapped Kubernetes operations assistant for Cloudera ECS, powered by a local LLM (Qwen3-8B on GPU / Qwen3-8B-GGUF Q4_K_M on CPU), LangGraph agentic loop, and LanceDB RAG.

⚠ This chatbot is read-only by default — it will never restart, delete, or modify cluster resources. Validated on Qwen3-8B running on GPU (HuggingFace Transformers) and Qwen3-8B-GGUF Q4_K_M running on CPU (llama-cpp-python). Using other self-hosted models may produce different or unexpected responses.

ECS AI Ops Chatbot

Each query runs through an agentic ReAct loop driven by the LLM. The LLM reads a Tool Selection Guide in its system prompt, picks the right tools, observes the results, and decides whether to chain further tool calls or synthesise a final answer. Each query is treated independently, there is no conversation history.

ECS Knowledge Bot

A knowledge base interface that retrieves relevant context from your ingested documents and synthesises a focused answer using the LLM — no live cluster calls. Best used for looking up runbooks, known issues, dos & don'ts, prerequisites, and past learnings.

Adjust the Max results slider if answers appear incomplete — increasing it retrieves more knowledge base entries per query.

Configuring the Cluster

Click ⚙ Settings → Kubeconfig tab. Paste your kubeconfig YAML and click Apply. The cluster host appears in the header on success. No restart needed — you can switch clusters at any time.

curl -s -X POST http://localhost:9000/api/kubeconfig \
     -H 'Content-Type: application/json' \
     -d "{\"kubeconfig\": \"$(cat ~/.kube/config | python3 -c 'import sys,json; print(json.dumps(sys.stdin.read()))')\"}"

Adjust LLM input/output limits at runtime via ⚙ Settings → LLM Input/Output, or via API:

Knowledge Base (RAG)

The assistant cross-references live cluster data with your runbooks, known-issues docs, and SOPs via LanceDB RAG. Both chatbots draw from the same LanceDB vector store.

Via the Settings UI

  1. Click ⚙ Settings → RAG Documents tab.
  2. Drop .md, .pdf, .txt, .xlsx, or .xls files onto the drop zone.
  3. Click Ingest. Previously ingested files with the same content are automatically skipped (deduplication is hash-based).
  4. To replace an existing file with updated content, tick Force re-ingest before clicking Ingest.

💡 All previously ingested files are listed in the Ingested Files panel at the bottom of the tab.

Via server-side directory

python3 app.py --ingest ./docs
python3 app.py --ingest ./docs --force   # force overwrite all

LLM — GPU vs CPU

The application auto-detects available GPUs at startup. Use the appropriate model for your hardware:

HardwareModelBackendResponse time
GPU (recommended)Qwen3-8BHuggingFace Transformers (bfloat16)10–30 s
CPU onlyQwen3-8B-GGUF Q4_K_Mllama-cpp-python (quantized)Several minutes

Set LLM_MODEL in the env file. If the path ends with .gguf the app automatically switches to the GGUF backend — no other config change needed.

# GPU
LLM_MODEL=/models/Qwen3-8B
NUM_GPU=1

# CPU (GGUF)
LLM_MODEL=/models/Qwen3-8B-GGUF/Qwen3-8B-Q4_K_M.gguf
NUM_GPU=0

Customising the System Prompt

The LLM's tool selection, multi-hop behaviour, and persona are controlled by config/system_prompt.txt. Key sections:

  • Tool Selection Guide — which tools to call for each query type
  • Multi-hop Reasoning — when to chain calls (e.g. OOMKilled → check quotas)
  • CRITICAL: Never Narrate — prevents the LLM from suggesting tools instead of calling them

Hot-reload after editing — no restart needed:

curl -s -X POST http://localhost:9000/api/reload-prompt

Security Notes

  • All typed K8s tools are read-only by design.
  • kubectl_exec is read-only by default. Set KUBECTL_ALLOW_WRITES=true to enable write ops.
  • Secret values are hidden by default — toggle in ⚙ Settings → Security. Preference persists per browser.
  • Never expose this service publicly — it has direct cluster read access.
  • Restrict the env file: chmod 600 env

📋 Prompt Guidelines for ECS AI Ops Chatbot

Tips and examples for getting the best responses from the ECS AI Ops Chatbot.

🔒 Non-intrusive: All live cluster queries are read-only — only GET operations are executed, never write or delete. Your cluster is safe.
💡 Golden rule: Always include a namespace when you know it — e.g. in cdp namespace — for faster and more accurate responses.

HOW THE CHATBOT DECIDES WHAT TO DO

Every question is independent — there is no conversation history. The chatbot determines your intent and takes one of two paths:

PathTriggered byWhat happens
Live cluster Any operational query — pods, namespaces, secrets, storage, ingress, resources, database… Runs a read-only tool against the live ECS cluster and synthesises the result.
🧠 Model knowledge Must add “don’t run it, just explain” if you want the model to describe what it would do. e.g. what command do you run to get pods in a namespace? Don’t run it, just explain. No tool is called. The model uses its trained knowledge to explain the approach. Without this instruction, operational queries will always execute against the live cluster.
💬 Conversational Greetings, identity questions, general chat Responds directly — no tool is called.

EXAMPLE QUERIES BY CATEGORY

Pods & Health

PODCLUSTER
list all pods
Lists all pods across all namespaces with their phase and ready status.
is the vault pod doing ok?
Checks pod phase, readiness, and recent restart history for any pod named vault.

Namespaces

CLUSTER
which namespace has the least pods?
Counts pods per namespace and ranks them.
list all namespaces with total pods
Returns every namespace with its pod count.

Networking & Ingress

CLUSTER
which namespace contains ingress port 443?
Searches all ingress resources across namespaces for TLS / port-443 exposure.

Storage

STORAGE
what storage type are pods in cdp namespace using?
Lists PVCs bound to pods in the cdp namespace, including storage class and access mode.

Secrets & Certificates

SECRET
what SSL certificates does the cdp namespace have?
Lists TLS/cert secrets in the cdp namespace. Enable Show Secret Values in Settings to decode.
what is the username and password for the SQL user for db-0 in cmlwb1 namespace?
Locates credentials from secrets in cmlwb1. Requires Show Secret Values to be enabled.

Resources & Compute

RESOURCE
list all statefulset in cdp namespace
what's the cpu and ram request for cdp-embedded-db pod in cdp namespace?
calculate CPU requests for all pods in longhorn-system namespace
Aggregates CPU and memory requests/limits for every pod in the namespace in one call.
which node has a GPU available and in use?
Inspects node labels and allocatable resources for GPU capacity and current allocation.

Database

DATABASE
get tables in db-0 of cmlwb1 namespace
Finds the DB pod, auto-discovers credentials, and runs SHOW TABLES.

Model Knowledge

HOW-TOGENERAL
⚠ These queries use model knowledge only — no live cluster tool is called and no cluster data is returned. Must add “don’t run it, just explain” when asking the model to describe what it would do.
explain how you access the database of a pod to get the table names, don't run it, just explain
Chatbot explains the exec_db_query flow: pod discovery → credential detection → SQL execution. No cluster tool is called.
what can you do?
Lists available capabilities. No tool is called.
